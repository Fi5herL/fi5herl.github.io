# n8n 與 GAS 架構功能差異比較

> **目的**：協助決策是否將現有 n8n 流程遷移至 Google Apps Script (GAS) 架構，聚焦於使用者可感受到的行為差異。  
> **範圍**：四大功能區塊——訂日期提醒、使用者下指令、檔案 LINE 自動備份、AI 總結。  
> **所有機密（Token / API Key）均存放於 GAS 指令碼屬性，不出現在程式碼或文件中。**

---

## 快速摘要表

| 功能區塊 | 等價度 | 主要差異 | 遷移風險 | 建議優先序 |
|---|---|---|---|---|
| 訂日期提醒 | ★★★★★ 95 % | 觸發時間 ±1 分鐘浮動 | 低 | 第一波 |
| 使用者下指令 | ★★★★☆ 90 % | 高峰慢 1–3 秒 | 低–中 | 第一波 |
| 檔案 LINE 自動備份 | ★★★☆☆ 75–85 % | 即時→準即時（10 秒–2 分鐘延遲） | 中 | 第二波 |
| AI 總結 | ★★★☆☆ 60–80 % | 即時→批次（排隊後回覆） | 中–高 | 第三波 |

---

## 一、訂日期提醒（訂日期提醒）

### 1-1 等價項目
- 定時查詢 Google Sheets 中的日期事件
- 於指定時間透過 LINE Push 訊息發送提醒
- 支援多筆事件、多組時間觸發

### 1-2 行為差異

| 面向 | n8n | GAS |
|---|---|---|
| 觸發精度 | Cron 準確（秒級） | Time Trigger 可能有 ±1 分鐘浮動 |
| 觸發管理 | 流程節點拖拉設定 | 程式碼設定 `ScriptApp.newTrigger(...)` |
| 觸發失敗通知 | n8n 內建 onError 節點 | 需自行實作錯誤捕捉 + Push 通知 |
| 使用者體感 | 準時收到提醒 | 準時（最多差 1 分鐘，通常感覺不到） |

### 1-3 風險／取捨
- **低風險**：GAS Time Trigger 穩定，Google 保證執行，只差精度約 ±1 分鐘。
- **注意**：GAS 每日執行時間有上限（免費版 6 分鐘/次、90 分鐘/日），提醒查詢量大時需注意批次設計。
- **無法即時反應**：若使用者在提醒前 1 分鐘才修改日期，GAS 該輪觸發可能已讀取舊資料（n8n 同樣有此限制，差異不大）。

### 1-4 建議遷移策略
1. 直接 1:1 移植現有 n8n 查詢邏輯到 GAS `sendDailyReminders()` 函式。
2. 設定 Time Trigger 每日 06:00 觸發（`ScriptApp.newTrigger('sendDailyReminders').timeBased().atHour(6).everyDays(1).create()`）。
3. **驗收標準**：連續 7 天在目標時間（±2 分鐘內）收到 LINE 提醒，且訊息內容與 Sheet 資料一致。

## 二、使用者下指令（使用者下指令）

### 2-1 等價項目
- 接收 LINE 群組 / 個人訊息中的文字指令（`/a`、`/d`、`/id`、`/help` 等）
- 查詢 Google Sheets 並回覆結果
- 寫入 / 刪除 Sheet 資料

### 2-2 行為差異

| 面向 | n8n | GAS |
|---|---|---|
| Webhook 接收 | n8n 內建 LINE trigger | `doPost(e)` 自行處理 |
| 指令解析 | Router 節點視覺化 | `commandRouter_()` 函式 |
| 回覆速度（一般情況） | 通常 1 秒內 | 通常 1–2 秒內（同等級） |
| 回覆速度（高峰） | 可能卡流程節點 | 先進 event_queue，worker 批次處理，最多延遲 1 分鐘 |
| 指令新增難度 | 新增節點即可 | 修改程式碼，但邏輯清晰可控 |
| 執行紀錄 | n8n 執行歷程 | 自行寫入 Sheet log |

### 2-3 風險／取捨
- **主要差異**：若採用 queue 架構，指令不再是「立即同步處理」，而是先寫入佇列、再由 worker 拾取。使用者可能在送出指令後等待最長 1 分鐘才收到回覆。
- **緩解方式**：對需要即時回應的指令（如 `/help`、`/id`）保留同步處理路徑；僅將寫入、刪除等操作放入 queue。
- **無視覺化**：維護人員需要習慣看程式碼而非流程圖。

### 2-4 建議遷移策略
1. **同步指令**（即時回覆）：`/help`、`/id`、查詢類 → 在 `doPost` 直接處理並回 reply。
2. **非同步指令**（寫入操作）：`/a`（新增）、`/d`（刪除）→ 先入 event_queue，worker 處理後 push 確認訊息。
3. 先做「指令路由骨架」讓所有指令可接收，再逐一補上各指令的業務邏輯。
4. **驗收標準**：每個指令在正常情況下 2 秒內收到回覆（或確認訊息），高峰時 1 分鐘內完成。

## 三、檔案 LINE 自動備份（檔案 LINE 自動備份）

### 3-1 等價項目
- 偵測 LINE 群組中的圖片、影片、音檔、一般檔案訊息
- 下載媒體內容並上傳至 Google Drive
- 在 Google Sheets 中記錄檔案連結與來源資訊

### 3-2 行為差異

| 面向 | n8n | GAS |
|---|---|---|
| 備份觸發時機 | 訊息進來立即觸發 | 訊息先進 media_queue，worker 每 1 分鐘批次處理 |
| 備份完成時間 | 通常數秒內 | 通常 10 秒–2 分鐘（依佇列長度） |
| 使用者得到 Drive 連結的時間 | 幾乎即時 | 稍後（worker 處理完後補上，或另外 push 通知） |
| 失敗重試 | n8n 內建重試 | 需自行實作 retry（30 s / 120 s / 300 s 退避） |
| 漏件風險 | 流程節點錯誤可能掉件 | 佇列架構不漏收，但需維護 dead-letter 機制 |
| 媒體類型支援 | 看 n8n 節點支援 | 需自行呼叫 LINE Content API（所有媒體型別均可） |

### 3-3 風險／取捨
- **最大差異**：從「即時備份」變成「準即時備份」，使用者送出檔案後不一定馬上在 Sheet 看到 Drive 連結。
- **可接受情境**：若使用者不需要馬上點連結，準即時（1–2 分鐘）通常可接受。
- **不可接受情境**：若使用情境是「傳完立刻要用連結」，需另外設計即時路徑或 push 完成通知。
- **媒體 Token 有效期**：LINE 媒體下載 token 有時效（通常數分鐘至數小時），queue 延遲過長可能導致下載失敗，需確認 worker 排程夠頻繁。
- **死信處理**：超過重試上限的訊息進 dead_letter sheet，需定期人工確認。

### 3-4 建議遷移策略
1. **先平行跑**：n8n 與 GAS 同時接收 Webhook，確認 GAS 備份結果與 n8n 一致後，再關閉 n8n 這條路徑。
2. 實作完整的 `media_queue`：欄位包含 `message_id`、`type`、`status`（pending/retry/done/dead）、`retry_count`、`next_retry_at`。
3. Worker 每 1 分鐘觸發，逾時保護設 50 秒，避免 GAS 執行超時。
4. 完成備份後，透過 LINE Push 通知使用者「檔案已備份，連結：…」（可選，依需求決定）。
5. **驗收標準**：傳送 10 張圖片、5 個檔案，2 分鐘內全數出現在 Drive 與 Sheet，dead_letter 為零。

## 四、AI 總結（AI 總結）

### 4-1 等價項目
- 分析 Google Sheets 中的訊息紀錄
- 呼叫外部 AI API（如 OpenAI、Gemini 等）產生摘要
- 將摘要寫入指定 Sheet 分頁

### 4-2 行為差異

| 面向 | n8n | GAS |
|---|---|---|
| 觸發方式 | 較偏即時觸發（指令或定時） | 建議批次化（每 5–10 分鐘，或指令後排隊） |
| 使用者等待感受 | 叫了較快回（數秒到數十秒） | 改為「排隊後完成通知」，非秒回 |
| API 呼叫控制 | n8n 節點設定 timeout | GAS 需自行設定 `UrlFetchApp` timeout（最長 30 s） |
| 多分表寫入 | n8n 多分支節點完整 | 需自行在程式碼中實作多表寫入邏輯 |
| 成本控制 | 較難批次合併請求 | 可批次合併多段對話再呼叫一次 AI，降低 token 費用 |
| 錯誤處理 | n8n onError 分支 | 需自行實作 try/catch + retry + 失敗通知 |
| GAS 執行時間限制 | 無此限制 | 單次最長 6 分鐘，長摘要需分批或使用非同步方案 |

### 4-3 風險／取捨
- **最大差異**：使用者體感從「叫了就快回」變成「叫了之後等通知」，需要 UX 層面的文案設計（例如：「已收到，正在生成摘要，完成後通知你」）。
- **執行超時**：GAS 單次執行上限 6 分鐘，若對話量龐大，單次 AI 呼叫可能逼近上限，建議設計分批邏輯。
- **成本優化機會**：GAS 批次架構可合併多次 AI 請求，實際 token 費用可能比 n8n 即時觸發更低。
- **API Key 安全**：透過 `PropertiesService.getScriptProperties().getProperty('AI_API_KEY')` 存取，不得硬寫在程式碼中。

### 4-4 建議遷移策略
1. **第一步：手動觸發 + 完成通知**：使用者輸入 `/summary` → GAS 收到後立即回「已排隊」→ worker 批次處理 → 完成後 Push 結果。
2. **第二步：定時自動摘要**：設定每 10 分鐘或每日特定時間自動產生摘要，不需使用者觸發。
3. 實作 `summary_queue`，欄位包含觸發者、時間範圍、狀態。
4. 設定 worker 的執行時間上限（`FORCE_SYNC_MAX_RUNTIME_SEC = 50`），超時自動切割下次繼續。
5. **驗收標準**：連續 5 次 `/summary` 指令，每次在 3 分鐘內收到 Push 摘要，且內容正確對應指定時間範圍。

## 最終建議

### 遷移優先序與理由

| 階段 | 功能 | 建議時間點 | 理由 |
|---|---|---|---|
| **Phase 1** | 訂日期提醒 + 使用者下指令 | 最先遷移 | 等價度高、風險低、可快速驗收 |
| **Phase 2** | 檔案 LINE 自動備份 | Phase 1 穩定後 | 需要 queue/retry 機制，先平行測試再切換 |
| **Phase 3** | AI 總結 | Phase 2 穩定後 | 差異最大，需要 UX 調整與批次設計，最後遷移 |

### 整體評估

- **建議遷移**：長期維護成本、成本可控性、程式碼版本化等優勢明顯，值得投入。
- **不建議一次全切**：分三波遷移，確保每波穩定後再進行下一波，風險可控。
- **保留 n8n 作為備案**：遷移期間不關閉 n8n，確認 GAS 運行 2 週無重大問題後再停用。
- **監控先行**：每波遷移前先建立基本監控（error rate、queue backlog、執行時間），才能在問題發生時快速判斷。

### 對使用者的承諾
- **訂日期提醒**、**指令操作**：使用者幾乎感覺不到差異。
- **檔案備份**：說明「系統升級後，檔案備份會在 2 分鐘內完成，完成後另行通知」。
- **AI 總結**：說明「輸入指令後，系統排隊處理，完成後推送結果，通常 3 分鐘內」。
